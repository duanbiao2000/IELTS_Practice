# Claude Code / Vibe Coding：什么算高质量？

这个问题问得非常关键——因为大多数人对 vibe coding 的理解停留在"让 AI 写代码跑通就行"，但**跑通和高质量之间的差距，跟雅思 Band 5 和 Band 8 之间的差距一样大**。

---

## 一、先定义"Vibe Coding"的本质

Vibe coding 不是"不懂代码让 AI 乱写"，而是：

> **你用自然语言精确描述意图，AI 生成实现，你判断质量并迭代。**

这意味着质量的瓶颈从来不在 AI 的代码能力，而在于：
- **你描述意图的精度**
- **你判断输出质量的能力**
- **你设计迭代流程的方法**

---

## 二、高质量 Vibe Coding 的六个维度

### 1. Prompt 工程质量——你给出的指令

| 低质量 | **高质量** |
|--------|-----------|
| "帮我写一个登录页面" | "用 Next.js 14 App Router + Server Actions 写登录，要求：邮箱+密码、rate limiting、CSRF 保护、错误信息不泄露用户是否存在" |
| 一次性给一个巨大需求 | **分层递进**：先架构 → 再核心逻辑 → 再边界处理 → 再测试 |
| 不提约束条件 | **明确约束**：性能预算、兼容性、依赖限制、团队规范 |
| 让 AI 自己决定所有事 | **给出决策但留出空间**："用 Zustand 做状态管理，你来决定 store 的拆分方式并解释理由" |

> **核心洞察：高质量的 prompt 不是更长，而是"约束恰到好处"——既不过度限制 AI 的能力，也不让它在关键决策上乱猜。**

---

### 2. 架构判断力——你能否看出"结构性问题"

这是 vibe coding 中**最需要人类专业度**的地方。

AI 写的代码经常有一个特征：**局部完美，全局混乱。**

| 你必须能判断的     | 具体含义                                  |
| ----------- | ------------------------------------- |
| **关注点分离**   | AI 是否把业务逻辑、数据获取、UI 渲染混在一个文件里？         |
| **抽象层次一致性** | 同一个函数里是否混合了高层意图和低层实现细节？               |
| **依赖方向**    | 核心业务逻辑是否依赖了框架细节？（违反依赖倒置）              |
| **状态管理**    | 状态是在正确的层级管理，还是到处 prop drilling 或全局滥用？ |
| **可扩展的接缝**  | 未来加需求时，是改一个文件还是要改 20 个文件？             |

```
❌ AI 经常做的事：
把所有逻辑写进一个 500 行的组件
→ 当下能跑
→ 第三次迭代时整个文件变成不可维护的泥球

✅ 高质量的做法：
你在第一次交互就告诉 AI 目录结构和模块边界
→ 每次迭代只动一个模块
→ 代码库始终可控
```

> **如果你不能画出系统的模块边界图，你就不具备 vibe coding 的高质量判断力。**

---

### 3. 代码审查能力——你能否看出 AI 的"典型错误模式"

Claude 和其他 LLM 有**系统性的弱点模式**，高质量的使用者必须知道：

| AI 的典型问题   | 为什么危险                         | **你该怎么检查**                     |
| ---------- | ----------------------------- | ------------------------------ |
| **幻觉 API** | 编造不存在的函数/参数                   | 对每个你不熟悉的 API 调用验证文档            |
| **过度工程**   | 简单需求搞出抽象工厂+策略模式               | 问自己："如果明天一个初级开发看这段代码，能30秒理解吗？" |
| **遗漏边界情况** | happy path 完美，null/空数组/并发全没处理 | 明确要求："列出所有可能失败的边界情况并处理"        |
| **安全漏洞**   | SQL 注入、XSS、密钥硬编码              | 永远让 AI 做一轮安全审查，再自己做一轮          |
| **性能陷阱**   | N+1 查询、不必要的重渲染、内存泄漏           | 对数据流路径做心理模拟："1000个用户同时请求会怎样？"  |
| **过时方案**   | 用 2021 年的写法（训练数据截止）           | 明确指定版本和文档链接                    |

> **高质量的标志：你能在 AI 输出的代码中，3分钟内指出哪些地方"看起来对但可能有问题"。**

---

### 4. 迭代策略——你如何驾驭多轮对话

这是大多数人做得最差的地方。

| 低质量迭代 | **高质量迭代** |
|-----------|---------------|
| 出了 bug → "帮我修这个 bug" | 出了 bug → **先分析 root cause** → "这个 bug 的根因是 X，请从 Y 层修复而不是在 Z 层打补丁" |
| 不断追加功能到同一个对话 | **超过10轮对话就开新会话**，带上精炼后的上下文 |
| 每次都给 AI 完整代码库 | **只给相关模块 + 接口定义**，减少噪声 |
| 让 AI 一次做完所有事 | **分阶段验收**：先跑通 → 加测试 → 加错误处理 → 性能优化 |
| 不保存好的中间状态 | **每个稳定版本 git commit**，可以随时回滚 |

> **核心原则：Vibe coding 的高质量不在于每次对话的输出质量，而在于你管理整个对话链条的能力。**

---

### 5. 测试与验证——你如何证明"它真的对"

| 验证层次 | **高质量做法** |
|---------|---------------|
| **功能正确性** | 让 AI 写测试，但**你来定义测试场景**（AI 不知道你的业务边界） |
| **类型安全** | 用 TypeScript 严格模式，把类型定义当作 AI 的"合同" |
| **集成验证** | 不只是单元测试——**让 AI 写 integration test，验证模块间的交互** |
| **手动验证** | 关键路径永远自己点一遍，不要只相信测试通过 |
| **渐进式信任** | 新的 AI 生成代码 → 先在隔离分支 → 测试通过 → 再合入主干 |

```
最危险的情况：
AI 写了测试 + AI 写了代码 → 测试 100% 通过
→ 但测试本身就是错的（测试覆盖了 happy path 但漏了核心边界）

你的工作：审查测试本身的质量，而不只是看绿灯
```

---

### 6. 上下文管理——Vibe Coding 的"基础设施"

这是 Claude Code 特有的高质量维度：

| 做法 | **为什么重要** |
|------|---------------|
| **CLAUDE.md 文件** | 把项目规范、技术栈、命名约定写进去 → AI 每次对话自动加载上下文 |
| **明确的目录结构** | AI 生成的代码自动放到正确的位置，而不是全堆在根目录 |
| **接口先行** | 先定义 TypeScript interfaces/types → 再让 AI 填充实现 → 接口就是约束 |
| **Compact 对话** | 对话太长时主动 `/compact`，让 AI 总结要点再继续 |
| **多 Agent 协作** | 用不同的 Claude Code 窗口做不同任务（一个写代码，一个做 code review） |

## 三、一个判断框架：你的 Vibe Coding 在哪个段位？

| 段位 | 特征 | 类比 |
|------|------|------|
| **L1 - 生成器** | 能让 AI 写出跑通的代码 | 会用计算器做加法 |
| **L2 - 编辑器** | 能发现并修正 AI 输出中的明显问题 | 能检查计算器的输入是否正确 |
| **L3 - 架构师** | 能在 AI 写之前定义好模块边界和接口 | 能决定用什么公式解决什么问题 |
| **L4 - 系统设计者** | 能设计一套完整的 AI 辅助开发流程，让团队可复制 | 能设计整个计算系统 |
| **L5 - 质量守门人** | 能定义"什么算好"的标准，并建立自动化的质量门禁 | **这就是你说的"2人团队"里那个人** |

---

## 四、最关键的元认知

> ### **Vibe Coding 的高质量 = 你对"什么不该让 AI 决定"有清晰的判断。**

具体来说：

| AI 应该做的 | **你必须做的** |
|------------|---------------|
| 写具体实现代码 | **定义架构和模块边界** |
| 生成多种方案 | **选择哪个方案并解释为什么** |
| 写测试用例 | **定义测试场景和验收标准** |
| 重构代码 | **决定什么时候重构、重构到什么程度** |
| 解释技术方案 | **做业务权衡（速度 vs 质量 vs 成本）** |
| 写文档 | **定义文档的受众和详细度** |

---

## 五、回到你的原始框架

> "懂得把 AI 嵌入流程，让一套方法可以被团队复制和扩展"

在 vibe coding 中，这意味着那个人必须能：

1. **建立 CLAUDE.md + 项目规范**，让任何人加入都能直接 vibe code 而不偏离标准
2. **定义代码审查 checklist**，专门针对 AI 生成代码的典型问题
3. **设计分层验证流程**：AI 生成 → 自动化测试 → 人工审查关键路径 → 部署
4. **积累 prompt 模板库**，把团队验证过的高质量 prompt 模式沉淀下来
5. **知道什么时候停止 vibe coding**，切换到手写——当 AI 连续3轮没改对的时候，人类直接改往往更快

> **最终判断标准：你的 vibe coding 产出物，能否让一个不知道它是 AI 写的资深工程师 code review 时说"这代码写得不错"？**

如果能——那就是高质量。
